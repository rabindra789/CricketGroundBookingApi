Sports Ground Booking API

Production-ready REST API for booking sports grounds across cricket, football, badminton, tennis, and more. Built on ASP.NET Core, JWT authentication, SQL Server, and Entity Framework Core.

Base URL
http://SERVER_IP:5107/api/v1
Authentication

All protected endpoints require a Bearer token in the Authorization header.

Authorization: Bearer <your_jwt_token>
Tokens are issued on login. Store them in localStorage (web) or Secure Storage / Keychain (mobile). Never embed in URLs.
Standard response envelope
{
  "success": true,
  "data": { ... }
}
{
  "success": false,
  "message": "Descriptive error message"
}
Slot types
Type Description Conflict rule
Day Daytime slot Can coexist with Night
Night Nighttime slot Can coexist with Day
FullDay Entire day Blocks Day and Night for that date

Booking Lifecycle

Understanding the full lifecycle helps you build correct UI states.

Booking status flow

1
PendingPaymentBooking created, awaiting payment. Slot is held.
2
Paid → ConfirmedAfter successful payment the status becomes Confirmed automatically.
3
CancelledUser cancels via DELETE. Cancelled bookings are ignored in conflict checks.

Booking statuses

PendingPayment Confirmed Cancelled

Payment statuses

PendingPayment Paid Confirmed

Business rules

FullDay blocks Day & NightIf a FullDay booking exists on a date, no other slot type can be booked for that ground.
Day + Night coexistA Day and Night slot for the same ground/date can be booked independently.
No past-date bookingThe API rejects any booking for a date that has already passed.
No expired slotsSlots that have already started/passed cannot be booked even on the current date.
Cancelled = ignoredCancelled bookings do not count in conflict detection, allowing re-booking of released slots.
One review per bookingOnly confirmed bookings can leave feedback, and each booking allows exactly one review.

Error handling

All errors follow the standard envelope. HTTP status codes align with the error type.

Scenario HTTP message value
Missing / expired JWT 401 "Unauthorized"
Slot conflict 400 "Slot already booked for this date"
FullDay conflict 400 "A FullDay booking already exists for this ground and date"
Past-date booking 400 "Cannot book for a past date"
Expired slot 400 "This slot has already expired"
Invalid rating 400 "Rating must be between 1 and 5"
Duplicate feedback 400 "Feedback already submitted for this booking"
Payment done already 400 "Payment already completed for this booking"
Not your booking 403 "You are not authorized to access this booking"
Resource not found 404 "Resource not found"
Always check success === false and display message to the user. Never rely solely on HTTP status codes in mobile clients.

Register

POST /api/v1/auth/register

Creates a new user account. No authentication required.

Request body
{
  "fullName": "Rohit Sharma",
  "email": "rohit@example.com",
  "phone": "9876543210",
  "password": "SecurePass@123"
}
Validation rules
Field Rules
email Must be unique, valid email format
password Min 8 chars and stored securely with bcrypt
phone Must be unique
Success response 200
{
  "success": true,
  "message": "registration successful",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "role": "User",
    "expiresAt": "2026-05-17T17:00:00Z"
  }
}
After registration, use the returned token for protected endpoints in the Authorization header.

Cancel Booking

DELETE /api/v1/Booking/{id}

Cancels the specified booking. Only the booking owner can cancel. Cancelled bookings are ignored in future conflict checks.

Success response 200
{
  "success": true,
  "message": "Booking cancelled successfully"
}
Show a confirmation dialog before calling this endpoint. Cancellation may not be reversible depending on business policy.

Frontend integration guide

Authentication flow

1
Register / LoginStore the JWT token securely. Set as default header for all API calls.
2
Protected routesCheck for token presence on route change. If absent, redirect to login.
3
Token expiryOn 401 response, clear token and redirect to login.

Booking flow

1
Browse groundsGET /api/v1/grounds → render cards
2
Select date on calendarGET /api/v1/Booking/availability?groundId=&date= → disable booked slots
3
Choose slot & addonsGET /api/v1/slots/ground/{id} + GET /api/v1/addons → show pricing
4
Create bookingPOST /api/v1/Booking → receive bookingId (PendingPayment)
5
PayPOST /api/v1/payments/pay/{bookingId} → booking becomes Confirmed
6
Confirm screenShow transactionId, summary card, and optional feedback CTA

JavaScript fetch example

Login and store token
const res = await fetch('http://SERVER_IP:5107/api/v1/auth/login', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ email, password })
});
const { success, data } = await res.json();
if (success) localStorage.setItem('token', data.token);
Authenticated request helper
const api = async (path, options = {}) => {
  const token = localStorage.getItem('token');
  const res = await fetch(`http://SERVER_IP:5107/api/v1${path}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${token}`,
      ...options.headers
    }
  });
  const json = await res.json();
  if (!json.success) throw new Error(json.message || 'API error');
  return json.data;
};

// Usage examples:
const grounds = await api('/grounds');
const booking = await api('/Booking', {
  method: 'POST',
  body: JSON.stringify({ groundId:1, slotId:2, bookingDate:'2026-06-01', slotType:'Night', addonIds:[1] })
});
const confirmed = await api(`/payments/pay/${booking.id}`, { method:'POST' });
